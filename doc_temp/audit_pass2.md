## Новые расхождения (не найденные в Прогоне 1)

### Р2-01: Математическая ошибка в 03-planets.md — «Крупная = 36 гексов»

- **Где:** 03-planets.md §3.2
- **Описание:** Документ утверждает: «Крупная | 36 | 1 + 6 + 12 + 18 (три полных кольца + начало четвёртого)». Но 1+6+12+18 = **37**, а не 36. Кроме того, «Карликовая = 6 (1+5)» — кольцо 1 в гекс-сетке всегда содержит ровно 6 гексов, а не 5. Частичные кольца не образуют корректную гексагональную сетку.
- **Влияние:** Вся система размеров 6/12/24/36 из 03-planets.md геометрически некорректна. Код использует правильные числа 19/37/61/91/127 (соответствующие полным кольцам гекс-сетки).

### Р2-02: Физическая ошибка в формуле гравитации 03-planets.md

- **Где:** 03-planets.md §2.2
- **Описание:** Документ даёт формулу: `gravity = (radius / earthRadius)² × (density / earthDensity)`. Правильная формула для поверхностной гравитации: `g = G·M/r² = (4/3)π·G·ρ·r`, т.е. gravity = (radius/earthRadius) × (density/earthDensity), **без квадрата** на radius.
- **Проверка**: Марс (r=3390 км, ρ=3.93 г/см³, реальная g=0.38):
  - По формуле документа: (3390/6371)² × (3.93/5.51) = 0.283 × 0.713 = **0.202g** ❌
  - По правильной формуле: (3390/6371) × (3.93/5.51) = 0.532 × 0.713 = **0.379g** ✓
- **Самопротиворечие**: Собственные данные документа о гравитации газовых гигантов (1.5–3.0g) согласуются с **правильной** формулой, но не с формулой документа. Газовый гигант r≈60000 км, ρ=1.5: формула документа даёт 24.1g, правильная формула — 2.56g.
- **Влияние:** Критическое. Любой код, реализующий формулу из документа, даст завышенную гравитацию для больших планет и заниженную для малых. Это ломает всю систему бонусов/штрафов гравитации.

### Р2-03: Сломанная цепочка рецептов — 11 из 15 руд не имеют рецептов переплавки

- **Где:** recipes.ts, RAW_ORES
- **Описание:** В коде 15 видов сырья (Fe-ore, Si-ore, Ti-ore, Cu-ore, Al-ore, C-ore, Ni-ore, W-ore, U-ore, Co-ore, Au-ore, Pt-ore, Cr-ore, Li-ore, S-ore), но рецепты переплавки есть только для 4: Fe-ore, Si-ore, Ti-ore, Cu-ore. Остальные 11 руд добываются шахтой, но **никогда не могут быть превращены в чистые элементы**.
- **Критические разрывы цепочки**:
  - **Al-ore → Al**: Нет рецепта. Al необходим для «Титановый сплав» (Ti+Al), а титановый сплав — для «Бронеплита».
  - **C-ore → C**: Нет рецепта. C необходим для «Стальной сплав» (Fe+C) — фундамент всей промышленности.
  - **W-ore → W**: Нет рецепта. W необходим для «Бронеплита» (titanium_alloy+W).
  - **Au-ore → Au**: Нет рецепта. Au необходим для «Микрочип» (silicon_crystal+Au).
  - **O**: Даже не является рудой — атмосферный элемент, но газовый экстрактор не производит конкретные элементы.
- **Влияние:** Невозможно произвести: сталь → корпусной элемент → корабль; титановый сплав → бронеплита; микрочип → лазер. Вся цепочка от сырья до корабля **разорвана**. Игрок может добывать руды, но не может их использовать.

### Р2-04: Формула температуры планеты не содержит преобразования единиц

- **Где:** 03-planets.md §2.3
- **Описание:** Документ даёт формулу: `T_planet = T_star × sqrt(R_star / (2 × D)) − 273.15`, но не указывает единицы R_star и D. Если R_star в R☉, а D в АЕ, необходим коэффициент перевода (1 R☉ = 0.00465 АЕ).
- **Проверка**: Земля (T=5778K, R=1 R☉, D=1 АЕ):
  - Без коэффициента: 5778 × sqrt(1/2) − 273 = **3815°C** ❌
  - С коэффициентом: 5778 × sqrt(0.00465/2) − 273 = **5°C** ✓
- **Влияние:** Без указания единиц и коэффициента перевода формула даёт абсурдные результаты. Программист, реализующий формулу как есть, получит температуры планет в тысячи градусов.

### Р2-05: Газовый экстрактор на газовом гиганте — противоречие между 03-planets.md и 04-buildings.md

- **Где:** 03-planets.md §1.2.6 vs 04-buildings.md §2.3
- **Описание:** 03-planets.md прямо говорит: «Застройка поверхности — Невозможна (0 гексов)». 04-buildings.md §2.3 содержит строку «Газовый гигант» в таблице эффективности газового экстрактора с efficiency=1.0, что подразумевает возможность строительства на газовых гигантах. Код (buildings.ts) разрешает gas_extractor на huge-планетах, к которым относится gas_giant.
- **Влияние:** Три источника дают три разных ответа: 03-planets.md = «нельзя», 04-buildings.md = «можно и эффективно», код = «можно». Документация внутренне противоречива.

### Р2-06: Ядерный реактор: базовая мощность 25 (док) vs 10 (код)

- **Где:** 04-buildings.md §8.2 vs economy/engine.ts
- **Описание:** 04-buildings.md указывает разные base_output для разных типов электростанций: солнечная=10, ядерная=25, термоядерная=100. В коде (engine.ts) оба типа используют `production += 10 * levelMult` — единая формула для всех.
- **Проверка**:
  - Ядерный реактор ур.1: док = 25×1×1.0 = 25, код = 10×1.2 = **12**
  - Ядерный реактор ур.5: док = 25×5×1.0 = 125, код = 10×2.0 = **20**
  - Ядерный реактор ур.10: док = 25×10×1.0 = 250, код = 10×3.0 = **30**
- **Влияние:** Ядерная электростанция в коде производит менее половины документированной мощности. Баланс экономики нарушен: ядерный реактор не даёт значимого преимущества над солнечной станцией, хотя стоит дороже (U-топливо).

### Р2-07: Код не различает типы электростанций в расчёте энергии

- **Где:** economy/engine.ts §recalcEnergyBalance
- **Описание:** Функция `recalcEnergyBalance` проверяет `buildingDef.category === 'energy'` и применяет `production += 10 * levelMult` для ВСЕХ энергетических зданий. 04-buildings.md описывает 5 типов электростанций (солнечная, ядерная, термоядерная, геотермальная, антиматериальная) с принципиально разными формулами и базовыми мощностями (10/25/100/15/500). Код игнорирует тип источника энергии.
- **Влияние:** Термоядерный реактор (doc: base=100) будет производить столько же, сколько солнечная панель (doc: base=10). Разница в 10× потеряна.

### Р2-08: Весовые коэффициенты местности газового гиганта не суммируются в 100%

- **Где:** planet-types.ts
- **Описание:** terrainWeights для gas_giant: `{plains: 40, ...остальные 0}`. Сумма = 40, а не 100. Хотя weightedChoice нормализует веса (эффективно все 40/40 = 100% plains), это нарушает инвариант «веса суммируются до 100», ожидаемый от остальных планет.
- **Влияние:** Если логика рендеринга или UI использует веса без нормализации (ожидая сумму=100), отображение будет некорректным. Более того, газовые гиганты вообще не должны иметь гексов.

### Р2-09: 03-planets.md фиксирует один размер на тип планеты, а код назначает только один

- **Где:** 03-planets.md §1.1 vs planet-types.ts vs generator.ts
- **Описание:** 03-planets.md §1.1 указывает, что каждый тип планеты может иметь несколько размеров: Скалистая = «Малая / Средняя / Крупная», Вулканическая = «Малая / Средняя / Крупная» и т.д. Однако в planet-types.ts каждый тип имеет фиксированный размер (rocky=medium, volcanic=small, ice=small и т.д.), а generator.ts добавляет лишь ±1 вариацию от базового. Газовый гигант всегда huge (127 гексов), хотя по документации должен иметь 0 гексов.
- **Влияние:** Все планеты одного типа генерируются с одинаковым базовым размером, что уменьшает разнообразие. Нет реализации множественных размерных классов для одного типа планеты.

### Р2-10: Рандомизация параметров звёзд не покрывает документированные диапазоны

- **Где:** generator.ts vs 02-stars.md §1.1
- **Описание:** Код использует `starDef.mass * (0.8 + rng.nextFloat() * 0.4)`, что даёт диапазон ±20% от базового значения. Но документированные диапазоны гораздо шире:
  - M-type: L = 0.001–0.08 L☉ (80× диапазон). Код: 0.005 × [0.8–1.2] = 0.004–0.006 (1.5× диапазон)
  - O-type: L = 30000–1000000 L☉ (33× диапазон). Код: 200000 × [0.8–1.2] = 160000–240000 (1.5× диапазон)
  - B-type: M = 2.1–16 M☉ (7.6× диапазон). Код: 10 × [0.8–1.2] = 8–12 (1.5× диапазон)
- **Влияние:** Генерируемые звёзды гораздо менее разнообразны, чем описано. Для светимости, которая варьируется на порядки, линейная рандомизация ±20% физически некорректна (нужна логарифмическая шкала).

### Р2-11: Архитектура.md указывает только 6 типов планет (пропущена карликовая)

- **Где:** ARCHITECTURE.md §3.1.2

### Р2-12: Технологические ссылки в 04-buildings.md используют названия из ARCHITECTURE.md, а не из 06-research.md

- **Где:** 04-buildings.md §6.1–6.5 vs 06-research.md §2.1

Это названия веток из ARCHITECTURE.md §3.2.1 (Химия, Физика, Инженерия, Биология, Военные науки, Ксеноархеология). 06-research.md использует другие названия: «Вычислительные системы» (computing), «Энергетика» (power), «Материаловедение» (materials). ### Р2-14: Таблица эффективности газового экстрактора не покрывает все типы атмосферы

- **Где:** 04-buildings.md §2.3 vs 03-planets.md §2.4

### Р2-16: Формула HZ для белого карлика использует L=0.005, но код использует L=0.001

- **Где:** 02-stars.md §3.2 таблица vs star-types.ts
- **Описание:** 02-stars.md §3.2 приводит HZ для белого карлика при L=0.005 L☉ (среднее диапазона 0.001–0.01). Код задаёт white_dwarf.luminosity=0.001 (минимум диапазона). Это 5× разница.

- **Влияние:** Зона обитаемости вокруг белого карлика в коде в ~2.3× уже, чем в документации.

## Валидация формул и цепочек

### Р2-20: Зона обитаемости — формула верна, но не применяется в коде

- **Влияние:** Планеты генерируются на случайных орбитах, независимо от того, попадают ли они в HZ. Это делает определение «обитаемости» из документации бессмысленным в текущей реализации.

### Р2-23: Полная цепочка «от руды до ионного двигателя» — разорвана

- **Описание пошагово**:
  1. Fe-ore → Fe ✓ (smelt_fe)
  2. Si-ore → Si ✓ (smelt_si)
  3. Ti-ore → Ti ✓ (smelt_ti)
  4. Cu-ore → Cu ✓ (smelt_cu)
  5. **C-ore → C ❌** (нет рецепта; C нужен для steel)
  6. **Al-ore → Al ❌** (нет рецепта; Al нужен для titanium_alloy)
  7. **O → нет источника ❌** (атмосферный, но газовый экстрактор не производит конкретные элементы)
  8. Fe+C → steel ❌ (нет C)
  9. Ti+Al → titanium_alloy ❌ (нет Al)
  10. Cu+O → superconductor ❌ (нет O)
  11. steel+superconductor → engine_section ❌ (нет steel и superconductor)
  12. engine_section(2)+superconductor(1) → ion_engine ❌
- **Результат:** Цепочка **полностью разорвана** на 3 критических узлах (C, Al, O). Ионный двигатель невозможно произвести.

### Р2-24: Вероятности звёзд — сумма в документации ≈100.003%

- **Где:** 02-stars.md §1.1 + §2.1
- **Описание:** Основная последовательность: 0.003+0.1+0.6+3+7.5+12+76 = 99.203%. Специальные: 0.4+0.2+0.1+0.05+0.05 = 0.8%. Итого: 99.203+0.8 = **100.003%**. Технически превышает 100%, хотя и на пренебрежимую величину. Документ гласит «≈99.203%» и «≈0.8%», что корректно как приближение, но при точной нормализации весов это вызовет ошибку округления.
- **Влияние:** Минимальное. При реализации нужно нормализовать вероятности.

## Валидация звёздной физики

### Р2-25: Соотношение масса-светимость в коде физически корректнее, чем в документации

- **Описание:** Для звёзд главной последовательности L ∝ M^3.5 (приближённо).
  - G-type: M=1, L=1. L/M^3.5 = 1/1 = 1.0 ✓
  - M-type (код): M=0.3, L=0.005. M^3.5 = 0.005. L/M^3.5 = 1.0 ✓
  - M-type (док, среднее): M=0.25, L=0.02. M^3.5 = 0.0078. L/M^3.5 = 2.56 ❌ (завышена)
  - B-type (код): M=10, L=3000. M^3.5 = 3162. L/M^3.5 = 0.95 ✓
  - B-type (док, среднее): M=5, L=500. M^3.5 = 280. L/M^3.5 = 1.79 (допустимо)
- **Влияние:** Кодовые значения для M-звёзд физически корректнее, чем документированные средние. Документация использует средние значения, которые не согласуются со степенным законом масса-светимость. Это не ошибка документации как таковая (среднее по диапазону ≠ среднее по логарифмической шкале), но создаёт путаницу.

### Р2-26: Чёрная дыра: T=0 в коде может вызвать деление на ноль

- **Где:** star-types.ts
- **Описание:** black_hole: temperature=0, luminosity=0, radius=0. Формула температуры планеты `T_planet = T_star × sqrt(R_star / (2 × D))` при T_star=0 даёт T_planet=0 (−273.15°C), что безопасно арифметически. Но если формула использует luminosity в знаменателе где-либо, luminosity=0 вызовет деление на ноль.
- **Влияние:** Потенциальный runtime error. Рекомендуется guard: if luminosity ≤ 0 → skip temperature/energy calculations.
