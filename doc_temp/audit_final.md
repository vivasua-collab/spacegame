## Новые расхождения (Прогон 3 — не найденные в прогонах 1-2)

### Р3-01: Добыча производит чистые элементы, а не руды — цепочка крафта полностью разорвана

- **Где:** economy/engine.ts → processExtraction + galaxy/generator.ts → assignResourceDeposits + data/elements.ts
- **Описание:** Критическая логическая ошибка, не замеченная в прогонах 1-2. Цепочка крафта устроена так: руда (Fe-ore) → переплавка → чистый элемент (Fe) → стальной сплав → … → корабль. Однако:
  1. `assignResourceDeposits` создаёт залежи с `elementId` из `ELEMENTS`, т.е. 'Fe', 'Si', 'Ti' и т.д.
  2. `processExtraction` добавляет добытое напрямую как `planet.resources[deposit.elementId]`
  3. Результат: шахта немедленно производит **чистые элементы** ('Fe'), минуя стадию руды ('Fe-ore')
  4. Рецепты переплавки (`smelt_fe`) требуют 'Fe-ore' на входе, но 'Fe-ore' **никогда не производится** в нормальном игровом процессе
  5. Только `giveStarterResources` вручную даёт 'Fe-ore', 'Si-ore' и т.д. — вне тестирования руды недоступны
- **Влияние:** **Блокирующее.** Вся система переплавки мертва. Шахта → чистый Fe → стальной сплав работает, но только потому, что **обходит** стадию переплавки. Рецепты smelt_fe, smelt_si, smelt_ti, smelt_cu — неиспользуемы. Сырьевая экономика не работает как задумано.

### Р3-02: Проверка энергобаланса в производстве сравнивает с полной стоимостью, а списывает по-тиково

- **Где:** economy/engine.ts → processProductionQueue
- **Описание:** Строка 75: `if (planet.energyBalance < recipe.energyCost && recipe.energyCost > 0)` — проверяет, хватает ли **полной** энергии рецепта. Строка 85: `planet.energyBalance -= recipe.energyCost / item.total` — списывает **по-тиковую** долю. Несоответствие: для запуска производства нужна полная стоимость энергии, хотя расход постепенный. Для рецепта с energyCost=15 и time=400, при energyBalance=10, производство заблокировано, хотя 10 > 15/400 = 0.0375 (постатейного расхода).
- **Влияние:** Здания с дорогими по энергии рецептами не могут начать работу, даже если фактический по-тиковый расход покрывается балансом. Обратно: после первого тика, баланс может уйти в минус, т.к. проверка была одноразовой.

### Р3-03: Рецепт микрочипа назначен на chemical_plant вместо завода электроники

- **Где:** recipes.ts → make_microchip
- **Описание:** `buildingId: 'chemical_plant'`. Документация (04-buildings.md §6.1) определяет «Завод электроники» (electronics_plant) как отдельное здание. В коде такого здания нет (в BUILDINGS только 8 зданий, electronics_plant отсутствует). Микрочип, процессор, датчик, система наведения — все производятся на химзаводе, что противоречит дизайну.
- **Влияние:** Нельзя реализовать бонус смежности «завод компонентов + плавильня» из 04-buildings.md §5.1, т.к. все компоненты производятся на том же здании, что и базовые материалы.

### Р3-04: Нет валидации совместимости здания и размера планеты

- **Где:** economy/engine.ts → buildOnHex
- **Описание:** Функция `buildOnHex` проверяет: занят ли гекс, есть ли ресурсы. Но **не проверяет**, входит ли размер планеты в `buildingDef.size`. BuildingDef.mine.size = ['tiny','small','medium','large'], но код позволяет построить шахту на 'huge' (газовый гигант).
- **Влияние:** Здания строятся на планетах, где их быть не должно (шахта на газовом гиганте, верфь на карликовой планете).

### Р3-05: Экономический тик не масштабируется со скоростью игры

- **Где:** stores/game-store.ts → tick()
- **Описание:** При скорости x50: `gameState.time.tick += 50`, но `processEconomyTick` вызывается **один раз**. Результат: 50 игровых тиков проходят, но экономика обрабатывает только 1 тик. При x1 всё корректно, но при x5/x15/x50 экономика отстаёт от игрового времени в 5/15/50 раз.
- **Влияние:** На высокой скорости игры производство, добыча и потребление работают в 50 раз медленнее, чем должны. Это разрушает геймплей при ускорении.

### Р3-06: Нет валидации здания для рецепта при постановке в очередь

- **Где:** economy/engine.ts → enqueueProduction
- **Описание:** `enqueueProduction` не проверяет, существует ли на планете здание, указанное в `recipe.buildingId`. Можно поставить в очередь рецепт «Ионный двигатель» (buildingId: 'shipyard') на планете без верфи.
- **Влияние:** Очередь примет рецепт, который никогда не будет выполнен. Игрок не получит обратной связи.

### Р3-07: Газовый экстрактор не имеет механизма добычи атмосферных газов

- **Где:** economy/engine.ts → processExtraction
- **Описание:** processExtraction работает только с `hex.deposits` (ресурсные залежи на гексах). Газовый экстрактор — здание категории 'extraction', но у него нет специальной логики для добычи из атмосферы. Нигде в коде не определяется, какие газы доступны на планете с атмосферой, и каков их состав.
- **Влияние:** Газовый экстрактор добывает из залежей (как шахта), а не из атмосферы. H, He, N, O (атмосферные элементы) никогда не производятся газовым экстрактором. Это усугубляет разрыв цепочки крафта (O нужен для сверхпроводника).

### Р3-08: Кварц (quarry) не производит неметаллических руд

- **Где:** economy/engine.ts → processExtraction
- **Описание:** Карьер (quarry) — здание категории 'extraction', но processExtraction не различает типы добывающих зданий. Все здания добычи работают одинаково: извлекают из hex.deposits. Нет логики, ограничивающей шахту металлами, а карьер — неметаллами.
- **Влияние:** Разница между шахтой и карьером — только в стоимости и бонусах местности. Оба добывают из одних и тех же залежей без специализации.

### Р3-09: Black hole: mass=10 (одиночное значение) не покрывает документированный диапазон 3-50 M☉

- **Где:** star-types.ts + 02-stars.md §2.6
- **Описание:** Код задаёт black_hole.mass=10. С рандомизацией ±20% диапазон: 8-12 M☉. Документация: 3-50 M☉. Фактический охват: ~2% от документированного диапазона.
- **Влияние:** Все чёрные дыры в игре практически одинаковы. Нет звёздных чёрных дыр (3-10 M☉) vs промежуточных (50+ M☉).

### Р3-10: FleetOrder не включает 'trade' и 'mine'

- **Где:** types.ts → FleetOrder
- **Описание:** `type: 'move' | 'patrol' | 'colonize' | 'attack' | 'defend'`. Документация описывает приказы: торговать, добывать на астероидах, исследовать — ни один из них не представлен.
- **Влияние:** Невозможно реализовать торговые флоты (Синдикат), добывающие флотилии, научные корабли.

### Р3-11: ARCHITECTURE.md §3.1.3 указывает 50 элементов, но §5.1 — «20 ключевых»

- **Где:** ARCHITECTURE.md §3.1.3 vs §5.1
- **Описание:** §3.1.3: «Система из **50 элементов**» и таблица с 40 элементами (10+10+10+10). §5.1: «Начать с 20 ключевых». Противоречие: если §3.1.3 описывает MVP, то 50 — слишком много; если полную версию — нужно явно указать.
- **Влияние:** Неясно, что именно входит в MVP.

### Р3-12: Текстовый комментарий в elements.ts утверждает «20 стартовых», но элементов 19

- **Где:** elements.ts строка 2
- **Описание:** Комментарий: «Определения элементов — 20 стартовых для MVP». Фактически 19 элементов. Это уточнение Р1-12: файл сам себя документирует неверно.
- **Влияние:** Вводит в заблуждение.

## Новые находки Прогона 3 (подробно)

### Р3-01: Добыча производит чистые элементы, а не руды

Это **самое критическое** расхождение, обнаруженное только в Прогоне 3. Прогоны 1-2 нашли отдельные симптомы (Р1-25: руды без ElementDef, Р2-03: нет рецептов переплавки), но не увидели корневую причину: **система добычи принципиально не производит руды**.

**Корень проблемы:**

```
generator.ts: assignResourceDeposits()
  → deposit.elementId = element.id          // 'Fe' (из ELEMENTS)

engine.ts: processExtraction()
  → planet.resources[deposit.elementId] += ...  // кладёт 'Fe', не 'Fe-ore'

recipes.ts: smelt_fe
  → inputs: { 'Fe-ore': 10 }                // требует 'Fe-ore', которого нет
```

**Исправление:** Либо (A) deposits должны производить руды ('Fe-ore'), либо (B) рецепты переплавки должны принимать чистые элементы ('Fe') на вход. Вариант A логичнее: разделяет сырьё и продукты переработки.

### Р3-05: Экономический тик не масштабируется

```typescript
// game-store.ts → tick()
gameState.time.tick += gameState.speed;        // += 50 при x50
processEconomyTick(allPlanets, queues);         // вызывается 1 раз
```

При x50 за 1 реальную секунду проходит 50 игровых секунд, но экономика обрабатывает 1 тик. Фикс: вызывать processEconomyTick `speed` раз, или добавить параметр множителя.
